# 実装と docs の同期ポリシー

## 目的

この文書は、研究室実装のコード、仕様書、設計文書、issue を常に同期させるための運用ルールを定める。

## 基本方針

1. 実装を変更したら、同じ change で関連 docs を更新する
2. 実装済みの事実は `docs/specifications/` に記録する
3. 設計上の責務境界や公開面が変わった場合は `docs/architecture/` も更新する
4. アルゴリズム固有の判断が変わった場合は `docs/algorithms/` も更新する
5. 作業単位の完了状態は `docs/issues/` に反映する

## 更新が必須になる変更

以下の変更では、docs 更新を必須とする。

- コンポーネント構成や責務分担の変更
- CLI や Python API の入出力変更
- シナリオ形式、設定ファイル、実行手順の変更
- アルゴリズムの処理フローや判断ロジックの変更
- テスト方針やテスト対象の変更
- upstream 由来コードの取り込み、改変、再配置

## 最低限の更新先

変更内容に応じて、最低限以下を更新する。

- 実装仕様: `docs/specifications/*.md`
- 構造設計: `docs/architecture/*.md`
- アルゴリズム別仕様: `docs/algorithms/*.md`
- 運用ルール: `docs/policies/*.md`
- issue 状態: `docs/issues/*.md`

## 例外

以下のように、振る舞いに影響しない変更は仕様書更新を省略してよい。

- コメントだけの修正
- タイポ修正
- フォーマット整形
- import 順整理などの無振る舞い変更

ただし、判断に迷う場合は docs も更新する。

## レビュー観点

実装を完了とみなす前に、少なくとも以下を確認する。

- 実装と仕様書の記述が矛盾していないか
- README と実行手順が現状のコマンドで再現できるか
- issue の完了条件が docs 上でも確認できるか
- placeholder 実装である場合、その事実が仕様書に明記されているか
